## 核心理念

**监控 → 可观测 → 智能运维**

从被动告警到主动洞察，让运维团队具备快速发现问题、定位根因、预测风险的能力。

| 维度 | 传统 Ops | AIOps |
|------|----------|-------|
| 触发方式 | 告警触发 — 故障已发生才知道 | 认知驱动 — 主动预防 |
| 处理方式 | 人工排查 — 依赖个人经验 | AI 推理 — 自动化分析 |
| 知识形式 | 个人经验、散落 | 结构化知识图谱、可复用 |
| 响应速度 | 分钟级（平均 30+ 分钟） | 秒级（5 分钟定位） |
| 扩展性 | 线性（人越多处理越多） | 指数（AI 可复制） |

## 系统架构总览

Observable Ops 采用五层架构体系，从数据接入到前端展示分层解耦：

```
┌─────────────────────────────────────────────────────────────────┐
│                    前端展示层 (React)                              │
│    仪表盘    知识中心    告警中心    工作台      Agent 对话       │
├─────────────────────────────────────────────────────────────────┤
│                    Agent 协作层 (6 类 Agent)                      │
│   Orchestrator  Monitor  Analyst  Investigator  Executor  Notifier│
├─────────────────────────────────────────────────────────────────┤
│                     AI 引擎层                                      │
│   接入引擎    检测引擎    推理引擎    决策引擎    学习引擎          │
├─────────────────────────────────────────────────────────────────┤
│                     存储计算层                                     │
│   时序数据库    知识图谱(Neo4j)    向量库    搜索引擎(Elasticsearch) │
├─────────────────────────────────────────────────────────────────┤
│                     数据接入层                                     │
│    Metrics(指标)  Logs(日志)  Traces(追踪)  Events(事件)  Alerts  │
└─────────────────────────────────────────────────────────────────┘
```

### 端到端数据流

```
数据输入 → 认知处理(接入/标准化/融合/知识图谱)
        → 智能分析(异常检测/关联分析/根因推理/趋势预测)
        → 响应行动(决策生成/执行规划/自动执行/效果评估)
        → 持续反馈(反馈学习/模型进化/知识丰富)
        → 回到认知处理（闭环迭代）
```

## Agent 协作体系

6 类专业 Agent 协同完成感知 → 分析 → 决策 → 执行全流程：

| Agent 类型 | 层级 | 职责 | 核心工具 |
|------------|------|------|----------|
| **Orchestrator** | 调度层 | 全局调度与决策、任务分解、资源协调 | 决策引擎、任务调度器 |
| **Monitor Agent** | 感知层 | 数据采集与异常检测、告警触发 | 指标查询、日志检索、追踪查询 |
| **Analyst Agent** | 分析层 | 根因分析与知识推理 | 知识图谱查询、因果推理、向量检索 |
| **Investigator Agent** | 分析层 | 深度调查与取证、日志挖掘 | 链路追踪、日志挖掘、Trace 分析 |
| **Executor Agent** | 执行层 | 自动修复与操作执行 | 脚本执行、API 调用、K8s 操作 |
| **Notifier Agent** | 通知层 | 通知与协调、审批流转 | 邮件、企微、短信、审批流 |

典型的故障处理时序：**用户/系统触发 → Orchestrator → Monitor（数据采集）→ Analyst（根因分析）→ Investigator（深度调查）→ Orchestrator（决策）→ Executor（执行）→ Notifier（通知）**

## 使用方式

### 开发模式（Docsify — 实时预览）

```bash
npm run dev
# 或
python3 serve/dev-server.py
# → http://localhost:4003/
```

开发模式直接从 `docs/` 加载 Markdown，支持文件修改后即时刷新，适合文档编辑与内容创作。

### 生产模式（静态 HTML）

```bash
# 构建单个文件
node build-one.js docs/01-intro/01-product-overview.md

# 批量构建全部文档
npm run build:all

# 启动静态站点
npm run serve
# 或
python3 serve/html-server.py
# → http://localhost:4003/
```

生产模式将 Markdown 通过 `marked` 渲染为独立 HTML 文件，输出到 `dist/` 目录，启动一个线程安全的静态文件服务器（支持 `BrokenPipeError` 等客户端断连保护）。

### 命令速查

| 命令 | 脚本 | 用途 | 端口 |
|------|------|------|------|
| `npm run dev` | `serve/dev-server.py` | Docsify 开发服务器 | 4003 |
| `npm run serve` | `serve/html-server.py` | 静态 HTML 生产服务器 | 4003 |
| `npm run build` | `build-one.js <path>` | 构建单个 MD → HTML | — |
| `npm run build:all` | `build-one.js` 批量 | 构建全部 27 篇文档 | — |

## 构建流程

```
┌─────────────┐     ┌─────────────────┐     ┌──────────────┐
│  Markdown   │────→│  build-one.js   │────→│   dist/      │
│  docs/*.md  │     │  (marked 渲染)   │     │    HTML 文件  │
└─────────────┘     └─────────────────┘     └──────────────┘
                           │
                           ↓
                    ┌─────────────────┐
                    │  template.html  │
                    │  (侧边栏 + 样式)  │
                    └─────────────────┘
```

构建过程：
1. `build-one.js` 读入 Markdown 文件 → `marked` 渲染为 HTML 正文
2. 读取对应章节的 `_sidebar.md` → 解析为 HTML 侧边栏（含折叠交互标签）
3. 注入 `template.html` 的 `{{CONTENT}}` / `{{SIDEBAR}}` / `{{TITLE}}` 占位符
4. 输出到 `dist/`，保持与 `docs/` 相同的子目录结构
